# Как сделать webhook'и безопасными для публичного приложения

Этот файл нужен перед деплоем на GitHub Pages. Адрес webhook'а в браузерном приложении нельзя сделать секретным: пользователь всегда может увидеть его в коде или в Network. Поэтому безопасность должна быть не в скрытии ссылки, а в правилах внутри Multy.

## Общий принцип

Webhook можно считать публичной дверью. Он должен отвечать только на ожидаемые запросы, не доверять данным из браузера и не возвращать ничего лишнего.

Что важно:

- `item` должен быть точным и разным для разных сценариев.
- Если нет user id из бота, доступ не выдавать.
- Платформу нельзя принимать "на веру" из браузера, если от этого зависит логика.
- Бонусы, приглашения и сохраненный прогресс нельзя просто принимать как готовую правду из frontend.
- Baserow token, настройки интеграций, пароли и экспорт графов с токенами не выкладывать в GitHub.

## Проверка подписки MAX

Текущий сценарий уже устроен правильно по базовой логике:

1. Приходит webhook.
2. Фильтр проверяет `item = max_channel_subscription`.
3. Multy проверяет подписку через свою интеграцию.
4. Ответ возвращает только `isSubscribed`.

Перед публикацией нужно добавить или проверить защитные условия:

- Если `max_user_id` пустой, отсутствует или выглядит как мусор, сразу вернуть:

```json
{ "ok": false, "isSubscribed": false, "reason": "missing_user_id" }
```

- Если пользователь не подписан:

```json
{ "ok": true, "isSubscribed": false, "reason": "not_subscribed" }
```

- Если подписан:

```json
{ "ok": true, "isSubscribed": true }
```

- Не возвращать внутренние поля Multy, данные профиля, названия CRM-полей, id интеграций и подробности ошибки.
- Не делать запись в Baserow в сценарии проверки подписки.

## Проверка подписки Telegram

Для Telegram лучше сделать отдельный webhook или отдельный сценарий с другим `item`, например:

```json
{ "item": "telegram_author_channels_subscription", "telegram_user_id": "..." }
```

Проверка должна быть строгой:

- пользователь должен быть подписан на оба канала: `adviceperm` и `gameneurons`;
- если подписка есть только на один канал, материалы не открывать;
- ответ должен быть таким же коротким, как в MAX:

```json
{ "ok": true, "isSubscribed": true }
```

или

```json
{ "ok": true, "isSubscribed": false, "reason": "not_subscribed" }
```

## Тренды и Baserow

Для колоды трендов отдельный риск: frontend сейчас может отправлять прогресс. Значит Multy должен проверять и чистить данные перед записью.

Что нужно сделать в сценариях загрузки и сохранения:

- `item` для загрузки и сохранения должны быть разными.
- `user_id` обязателен.
- `messenger` лучше фиксировать на стороне сценария: MAX-сценарий сам пишет `MAX`, Telegram-сценарий сам пишет `Telegram`.
- `collected_cards` принимать только как массив разрешенных id карт.
- Убирать дубли карт.
- Не принимать больше карт, чем существует в колоде.
- `bonus_cards` и `invited_friends` не увеличивать из обычного save-запроса.
- Бонусы за приглашения должен менять только отдельный сценарий реферальной логики.
- При загрузке отдавать только прогресс текущего пользователя, без чужих строк и служебных данных.

Пример безопасного ответа загрузки:

```json
{
  "ok": true,
  "exists": true,
  "collected_cards": [1, 4, 9],
  "bonus_cards": 0,
  "invited_friends": 0,
  "last_date": "2026-07-09"
}
```

## Что оставить в приложении

В коде приложения можно оставить URL webhook'ов, если Multy настроен как публичный endpoint с проверками выше.

В приложении важно:

- локальный обход подписки оставлять только для `file://`, `localhost`, `127.0.0.1`;
- на GitHub Pages без user id из бота показывать понятный экран: "Откройте приложение из бота";
- для MAX и Telegram использовать разные параметры запуска или разные URLs;
- не хранить токены Baserow, пароли Hostland, экспорты Multy с приватными ключами в репозитории.

## Минимальный тест перед деплоем

Проверить руками в Multy или через тестовые запросы:

- неправильный `item` не запускает нужный сценарий;
- запрос без user id возвращает `isSubscribed: false`;
- неподписанный пользователь получает `isSubscribed: false`;
- подписанный пользователь получает `isSubscribed: true`;
- попытка сохранить чужой или раздутый прогресс в трендах не записывает мусор в Baserow;
- frontend без user id на GitHub Pages не открывает материалы.

## Что не стоит делать

- Не импортировать измененный JSON графа вслепую поверх рабочего сценария.
- Не публиковать в GitHub экспорт Multy-графа, если внутри есть токены, id интеграций или настройки Baserow.
- Не считать URL webhook'а секретом.
- Не защищать материалы только скрытой кнопкой на frontend.

Сначала лучше настроить и проверить копии сценариев в Multy, а потом переключить приложение на финальные webhook URLs.
